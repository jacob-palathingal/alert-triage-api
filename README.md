# Alert Triage & Incident Summarization API

A backend service that ingests application logs and alerts, classifies them by severity, groups related events into incidents using time-window aggregation, and enriches each incident with a plain-English summary generated by the OpenAI API.

Built to demonstrate real-world backend system design: event ingestion, rule-based classification, stateful aggregation, AI enrichment as a non-critical layer, and a queryable REST API.

**Stack:** Python, FastAPI, PostgreSQL, Docker, OpenAI API, Railway

**Event ingestion pipeline:**
1. `POST /events` receives a log event with service, level, message, and timestamp
2. **Classifier** assigns severity (`high`, `medium`, `low`) using keyword and log-level rules
3. **Aggregator** groups the event into an open incident for the same service and severity within a 5-minute window, or creates a new incident if none exists
4. **Enricher** calls the OpenAI API on new incidents to generate a plain-English triage summary from the raw log message вҖ” if the API call fails, ingestion continues normally
5. Event and incident are persisted to PostgreSQL

## Endpoints

| Method | Endpoint | Description |
|--------|----------|-------------|
| `POST` | `/events/` | Ingest a log event |
| `GET` | `/incidents/` | List all incidents, newest first |
| `GET` | `/incidents/?severity=high` | Filter incidents by severity |
| `GET` | `/incidents/?service=auth-service` | Filter incidents by service |
| `GET` | `/incidents/{id}` | Get a single incident by ID |
| `GET` | `/health` | Liveness check |

## Example Usage

**Ingest a log event:**
```bash
curl -X POST https://your-railway-url/events/ \
  -H "Content-Type: application/json" \
  -d '{
    "service": "auth-service",
    "level": "ERROR",
    "message": "Database connection timeout after 30s",
    "timestamp": "2026-04-28T10:00:00Z"
  }'
```

**Response:**
```json
{
  "id": 1,
  "service": "auth-service",
  "level": "ERROR",
  "message": "Database connection timeout after 30s",

  "incident_id": 1,
  "timestamp": "2026-04-28T10:00:00Z",
  "created_at": "2026-04-28T10:00:01Z"
}
```

**Query incidents filtered by severity:**

## Severity Classification Rules

The classifier runs before any AI call and never fails:

| Condition | Severity |
|-----------|----------|
| Level is `CRITICAL`, `FATAL`, or `EMERGENCY` | `high` |
| Level is `ERROR` | `high` |
| Message contains: `timeout`, `connection refused`, `out of memory`, `disk full`, `crash`, `panic`, `unauthorized` | `high` |
| Level is `WARN` or `WARNING` | `medium` |
| Message contains: `retry`, `slow`, `degraded`, `high latency`, `rate limit` | `medium` |
| Everything else | `low` |

## Design Decisions

**Why rule-based classification first, not AI?**
Rules run in microseconds, never fail, and cost nothing. AI enrichment is reserved for generating human-readable summaries on new incidents only вҖ” not for every event. This keeps ingestion fast and reliable regardless of third-party API availability.

**Why does the enricher have a fallback?**
The OpenAI API is a non-critical enhancement. If it's unavailable or returns an error, ingestion continues and the incident summary stays null. This means a third-party outage can never take down the core pipeline.

**Why is `is_open` a string instead of a boolean?**
PostgreSQL unique constraints treat multiple `NULL` values as distinct, which lets us close an incident by setting `is_open = NULL` while enforcing that only one open incident exists per service and severity at a time. A boolean `false` would violate the constraint on the second closed incident.
