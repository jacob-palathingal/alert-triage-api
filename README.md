# Alert Triage & Incident Summarization API

A backend service that ingests application logs and alerts, classifies them by severity, groups related events into incidents using time-window aggregation, and enriches each incident with a plain-English summary generated by the OpenAI API.

Built to demonstrate real-world backend system design: event ingestion, rule-based classification, stateful aggregation, AI enrichment as a non-critical layer, and a queryable REST API.

**Stack:** Python, FastAPI, PostgreSQL, Docker, OpenAI API, Render

---

## Architecture

```
POST /events
     │
     ▼
┌─────────────┐     ┌──────────────┐     ┌─────────────┐     ┌──────────────┐
│  Classifier  │────▶│  Aggregator  │────▶│   Enricher   │────▶│  PostgreSQL  │
│             │     │              │     │              │     │              │
│ Rule-based  │     │ Configurable │     │ OpenAI API   │     │ events       │
│ severity    │     │ time-window  │     │ (non-critical│     │ incidents    │
│ assignment  │     │ grouping     │     │  fallback if │     │              │
│             │     │ into         │     │  unavailable)│     │              │
│             │     │ incidents    │     │              │     │              │
└─────────────┘     └──────────────┘     └──────────────┘     └──────────────┘
```

**Event ingestion pipeline:**
1. `POST /events` receives a log event — `timestamp` defaults to current UTC if not provided
2. **Classifier** assigns severity (`high`, `medium`, `low`) using keyword and log-level rules — never fails, no external calls
3. **Aggregator** groups the event into an open incident matching the same service and severity within the configured time window, or creates a new incident — handles concurrent creation safely via a DB-level unique constraint
4. **Enricher** calls the OpenAI API on new incidents to generate a plain-English triage summary from the raw log message — if the API call fails, ingestion continues normally with `summary` left null
5. Event and incident are committed to PostgreSQL together in a single transaction

## Endpoints

| Method | Endpoint | Description |
|--------|----------|-------------|
| `POST` | `/events/` | Ingest a log event |
| `GET` | `/incidents/` | List all incidents, newest first |
| `GET` | `/incidents/?severity=high` | Filter by severity |
| `GET` | `/incidents/?service=auth-service` | Filter by service |
| `GET` | `/incidents/{id}` | Get a single incident by ID |
| `GET` | `/health` | Liveness check |

## Example Usage

**Minimal payload — timestamp defaults to current UTC:**
```bash
curl -X POST https://alert-triage-api.onrender.com/events/ \
  -H "Content-Type: application/json" \
  -d '{
    "service": "auth-service",
    "level": "ERROR",
    "message": "Database connection timeout after 30s"
  }'
```

**Full payload with custom time window:**
```bash
curl -X POST https://alert-triage-api.onrender.com/events/ \
  -H "Content-Type: application/json" \
  -d '{
    "service": "payment-service",
    "level": "CRITICAL",
    "message": "Stripe API unreachable, transactions failing",
    "timestamp": "2026-05-07T10:00:00Z",
    "window_minutes": 1
  }'
```

**Response:**
```json
{
  "id": 1,
  "service": "auth-service",
  "level": "ERROR",
  "message": "Database connection timeout after 30s",
  "severity": "high",
  "incident_id": 1,
  "timestamp": "2026-05-07T10:00:00Z",
  "created_at": "2026-05-07T10:00:01Z"
}
```

**Query incidents filtered by severity:**
```bash
curl https://alert-triage-api.onrender.com/incidents/?severity=high
```

**Response:**
```json
[
  {
    "id": 1,
    "service": "auth-service",
    "severity": "high",
    "event_count": 4,
    "started_at": "2026-05-07T10:00:00Z",
    "ended_at": "2026-05-07T10:04:00Z",
    "summary": "The auth-service is experiencing database connectivity issues — investigate database server status and connection pool health first.",
    "created_at": "2026-05-07T10:00:01Z"
  }
]
```

## Event Payload Fields

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `service` | string | Yes | Name of the service emitting the log |
| `level` | string | Yes | Log level (ERROR, WARN, INFO, CRITICAL, etc.) |
| `message` | string | Yes | Raw log message |
| `timestamp` | datetime | No | Event time in UTC. Defaults to current UTC if omitted |
| `window_minutes` | integer | No | Incident grouping window in minutes. Default: 5, min: 1, max: 1440 |

## Severity Classification Rules

The classifier runs before any AI call and never fails:

| Condition | Severity |
|-----------|----------|
| Level is `CRITICAL`, `FATAL`, or `EMERGENCY` | `high` |
| Level is `ERROR` | `high` |
| Message contains: `timeout`, `connection refused`, `out of memory`, `disk full`, `crash`, `panic`, `unauthorized` | `high` |
| Level is `WARN` or `WARNING` | `medium` |
| Message contains: `retry`, `slow`, `degraded`, `high latency`, `rate limit` | `medium` |
| Everything else | `low` |

---

## Configurable Time Window

The `window_minutes` parameter controls how long an incident stays open after its last event. Events from the same service and severity arriving within the window are grouped into the same incident. Once the window expires, the next event opens a new incident.

Different services have different failure characteristics — a payment service might use a 1-minute window for fast triage, while a batch job service might use 30 minutes to account for normal processing delays. The default of 5 minutes suits most real-time services.

```
window_minutes=1:   tight grouping, incidents close fast
window_minutes=5:   default, suits most services
window_minutes=30:  for batch jobs or slow-moving failures
```

## Local Development

```bash
# 1. Clone the repo
git clone https://github.com/jacob-palathingal/alert-triage-api.git

# 5. Start PostgreSQL

uvicorn app.main:app --reload
```

## Design Decisions

**Why rule-based classification first, not AI?**
Rules run in microseconds, never fail, and cost nothing. AI enrichment is reserved for generating human-readable summaries on new incidents only — not for every event. This keeps ingestion fast and reliable regardless of third-party API availability.

**Why does the enricher have a fallback?**
The OpenAI API is a non-critical enhancement. If it's unavailable or returns an error, ingestion continues and the incident summary stays null. A third-party outage cannot take down the core pipeline.

**Why is `is_open` a string instead of a boolean?**
PostgreSQL unique constraints treat multiple `NULL` values as distinct, which lets us close an incident by setting `is_open = NULL` while enforcing that only one open incident exists per service and severity at a time. A boolean `false` would violate the constraint on the second closed incident.

**Why compare the window against event timestamps instead of wall clock time?**
The aggregator uses the incoming event's timestamp as the reference point, not `datetime.now()`. This means the window correctly reflects the time between events as recorded by the source service — not the time between API calls. It also makes the system testable with historical or replayed events.
